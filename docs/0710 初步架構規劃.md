# 管理員dashboard

* 管理員收到郵差送來的包裹：  
  * 新增一筆資料: 要送達的住址、存放於管理室的位子  
* 用戶收到Line 推送包裹通知後的用戶回覆須回傳給管理員  
* 管理員收到訊息，照完包裹後機器離開管理室的即時動作  
  * 出發前往門牌號  
  * 用戶取貨動作 \[已取貨、用戶退回、逾時退回\]  
  * 機器正在返回管理室  
* 機器人基礎狀態  
  * 當前狀態： Idle / Delivering / Charging / Error  
  * 電池電量： 顯示 % 數，若低於 20% 應以紅色高亮顯示。  
  * 當前位置： 顯示機器人目前的點位名稱 (例如：管理室、地址)  
* 艙門即時狀態 (H\_01 \~ H\_04)  
  * 門的狀態： 開啟 / 關閉。  
  * 載貨狀態： 空置 / 已鎖定（住戶剛下單）/ 已裝載（送貨中）。  
  * 對應房號： 若艙門已鎖定或裝載，需顯示對應的住戶住址。

💡 異常狀況防呆（如果有人說謊怎麼辦？）(應該先不用管，但放著)

當然，完全信任「人為點擊」會有風險。例如住戶按了「取貨完成」，但其實包裹還留在裡面；或者是管理員按了「出發」，但其實忘記放進去。為了防呆，你可以在管理員的 Dashboard 補上一個「**強制清空/校正**」的功能： 當機器人完成一趟任務回到管理室時，管理員可以透過 Dashboard 點擊「一鍵開啟所有艙門」來檢查。如果發現 H\_01 明明被住戶按了完成，但裡面還有東西，管理員就可以把東西拿出來退件，並在系統中進行狀態校正。

**推播失敗的處理**：用戶封鎖官方帳號、或推播API暫時失敗，要重試幾次、間隔多久、失敗之後誰要收到警訊 

* 任務佇列 (Task Queue)  
  * 待處理訂單： 住戶已發出請求，但機器人尚未回到管理室的清單。  
  * 進行中訂單： 正在配送途中的清單。  
  * 稍後處理訂單：用戶選擇稍後再取的訂單。  
  * 歷史紀錄： 今日已完成或失敗的配送紀錄。  
* 機器人上的確認關門、出發，可以用dashboard直接操控

| status |  | dashboard | 艙門狀態 |
| :---- | :---- | :---- | :---- |

| later | \[稍後再取\] | 稍後處理訂單 | 空 |
| pickup\_now | \[取貨\] | 待處理訂單 | 空 |

# Line

**LINE 通知及掃碼取貨: 到貨通知、呼叫機器人、用戶端機器人操作、掃碼取貨**

**系統元件**

|  | Line 官方帳號 | 住戶加入使用AMR | 用於通知用戶及互動 |
| :---- | :---- | :---- | :---- |
| API | Webhook | 接收住戶端事件 | 接收follow、message、postback等事件，並轉送後端處理 |
|  | Push/ Reply | 推播訊息 | 到貨通知、狀態更新等主動傳給用戶 |
|  | LIFF | 互動介面 | 住戶綁定門牌 |
|  | Rich Menu | 常駐選單 | 提供常用訊息入口給用戶 |
|  | Flex Message | 結構化訊息卡 | 到貨通知、機器人狀態的卡片訊息呈現 |
|  | Webhook Server | Line 與系統橋接 | 驗證簽章、呼叫API、回復訊息、解析事件 |

**Modules**

* **用戶綁定: 把 Line User ID和住戶資料(門牌、姓名)對應**  
  * 住戶加入社區官方Line 帳號---follow 事件  
  * Webhook 接收到事件後，回覆LIFF可以讓用戶進行綁定資料  
  * 住戶在LIFF 輸入門牌、姓名等進行身分紀錄及驗證  
  * 透過API，輸入資料會與內部資料連結

* **到貨通知: 管理員登記包裹到貨後，系統會透過Line Push API 推送訊息給用戶**  
  * 透過flex message 推播訊息  
1. **用戶操作Line 介面選擇取貨---postback**  
2. **用戶操作Line 介面選擇稍後再取---postback:** 先不做重推

* **呼叫機器人: 用戶按下取貨後**  
  * 通知管理員放貨，將包裹放入櫃位，管理員按下出發後櫃門鎖定  
  * 用戶端在機器人開始移動後應接收到回饋訊息(已開始送貨)---push  
  * 機器人抵達後，用戶端接收\[已抵達，請於10分鐘內取貨\]通知，並於Line介面提供\[暫時無法取貨\]按鈕。---push

* **住戶取貨**  
1. **QR Code驗證取貨:** 用戶透過Line 掃描機器人螢幕上的QR Code，掃描結果回傳後端驗證---LIFF，驗證通過後艙門開啟  
2. **臨時無法取貨:** 住戶在Line 上操作包裹退回---pushback，Line 推播已退回通知---push  
3. **逾時未領(ex. 10 min):** 系統判定逾時，機器自動退回，Line 推播已退回通知---push

   

* **機器返回**  
1. 用戶在Line 介面按下取貨完成，機器關門返回管理室  
2. 若未取貨成功，機器返回後Line 可推播\[已退回，請洽管理室\]通知

# 送貨機器人

流程: (簡單、一個貨物、一個住戶)

1. 管理員收到貨、住戶收到通知 \-\> 住戶點選**取貨**  
2. 機器人: 管理室 \-\> 打開艙門  
3. 管理員放入**指定貨品**進入**指定艙門** \-\> 按下送貨 \-\> 關閉艙門  
4. 機器人前往住戶住址門口 \-\> 住戶掃描QR Code \-\> 打開艙門  
5. 住戶按下取貨完成 \-\> 機器人關閉艙門 \-\> 回到管理室

[API function整理](https://docs.google.com/spreadsheets/d/1ZFSpy3VwWnXTl6uAS6VDN7OeFCF9bXPXwQd7LoymdtQ/edit?usp=sharing) (目前存在的)  
常用: custom\_call2、control\_doors

前置作業:

* 地點設定--地圖、管理室、住戶住址  
* 資料: 單號-住戶住址-住戶-收貨位置編號-放置艙門位置編號  
* 輪詢: 一定時間(3秒?)就呼叫一次state\_get

| 步驟 | 情境 | 中控後台系統採取的動作 / 呼叫的普渡API | 備註與狀態監控 |
| ----- | ----- | ----- | ----- |
| 1 | 住戶點選取貨 | 1\. 收到請求，後台檢查機器人是否IDLE (state\_get) 2\. 後台為此訂單分配一個可用艙門 (例如 H\_01) 並寫入資料庫 3\. 呼叫 custom\_call 讓機器人前往管理室 | 機器人若在充電，需先中斷或確認電量是否足夠。 |
| 2 | 機器人前往收貨點並打開艙門 | 1\. 後台輪詢 state\_get 發現狀態為 Arrive 2\. 後台呼叫 control\_doors 打開預定分配的 H\_01 艙門 3\. 推播通知到管理員dashboard：「機器人已抵達，請將包裹放入 H\_01」 | 可搭配 screen\_set 在機器人螢幕顯示。 |
| 3 | 管理員放貨、關門出發 | 1\. 管理員放好貨，在平板按下「確認出發」 2\. 後台呼叫 control\_doors 關閉 H\_01 3\. 後台呼叫 custom\_call，指令機器人前往指定點位 4\. 用戶若在運送過程點選取消，呼叫custom\_cancel，並custom\_call回管理室 |  |
| 4 | 機器人前往住戶門口並打開艙門 | 1\. 後台輪詢 state\_get 發現狀態為 Arrive 2\. 後台查資料庫得知這趟是 H\_01 的貨 3\. 後台傳送 LINE 通知住戶：「包裹已送達房門口，請掃描 QR Code 取貨」 | 可使用 screen\_set 設定住戶取件提示。  |
| 5 | 住戶取貨、機器人返航 | 1\. 住戶走到門口，用 LINE 掃描 QR Code 取貨， 2\. 後台收到取貨訊號，呼叫 control\_doors 打開 H\_01 3\. 住戶在 LINE 點擊「取貨完成」 4\. 後台呼叫 control\_doors 關閉艙門 5\. 後台呼叫 custom\_call (或 recharge) 讓機器人返回管理室 10 min 逾時未領： 1\. custom\_cancel剛剛QR Code的動作 2\. custom\_call (或 recharge) 讓機器人返回管理室 3\. 打開同一艙門讓管理員拿出包裹 | 「取貨完成」後，後台將訂單狀態更新為「已簽收」。 |
